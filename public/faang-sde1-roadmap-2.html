🧠 DSA — Complete Pattern Library

Every pattern you need. Understand the pattern, not the problem. Then any new problem becomes a variation of something you already know.

📊Arrays & Strings — All Patterns
  • ★Two Pointer — opposite ends: sorted pair sum, container water
  • ★Two Pointer — same direction: remove duplicates, move zeros
  • ★Sliding Window fixed: max avg subarray of size K
  • ★Sliding Window variable: longest substring no repeat
  • →Prefix Sum: range sum queries O(1) after O(n) build
  • →Difference Array: range update O(1), query O(n)
  • →Kadane's: max subarray — track current and global max
  • →Dutch National Flag: 3-way partition, sort colors
  • →In-place reversal: strings, rotate array by k
  • →Trapping Rain Water: two pointer and stack approaches
  • →Product array except self — no division trick
  • →Next permutation — find next lexicographic order
  • →Binary search on answer: minimize max, painter's partition
  • →KMP string match: failure function, O(n+m)
  • →Z-function: all positions string appears as prefix
🌳Trees — Every Pattern
  • ★BFS level-order: queue, level-by-level processing
  • ★DFS preorder iterative: explicit stack, push right first
  • ★DFS inorder iterative: go left till null, pop and go right
  • ★LCA: recurse — if both nodes found in subtrees, current is LCA
  • →Max depth: max(left,right)+1 — simplest recursion
  • →Diameter: max of (left_depth+right_depth) at each node
  • →Balanced check: height returns -1 if unbalanced
  • →Path sum III: prefix sum map tracks running totals
  • →Right side view: last node of each BFS level
  • →Construct from preorder+inorder: root=pre[0], split inorder
  • →Serialize/Deserialize: preorder with null markers
  • →Morris traversal: O(1) space using threading
  • →Trie insert/search/prefix: each node has 26 children + isEnd
  • →Segment tree: range sum/min with lazy propagation
  • →Fenwick tree (BIT): point update, prefix query O(logn)
🕸️Graphs — Every Algorithm
  • ★BFS: shortest path unweighted, level-by-level
  • ★DFS: connected components, cycle detect directed/undirected
  • ★Topological sort Kahn's: in-degree array, queue zero-in
  • ★Topological DFS: post-order, reverse = topo order
  • ★Union-Find: path compression + rank — near O(1)
  • →Dijkstra: min-heap (dist, node), greedy shortest path
  • →Bellman-Ford: relax all edges V-1 times, detect neg cycle
  • →Floyd-Warshall: dp[i][j] = min via k, all pairs O(V³)
  • →Prim's MST: min-heap picks minimum crossing edge
  • →Kruskal's MST: sort edges, union-find for cycle check
  • →Bipartite: 2-color BFS, fail if same color adjacent
  • →Word Ladder: BFS on implicit graph, wildcard hashing
  • →Bridges: Tarjan's low-link values
  • →SCC Kosaraju: DFS → reverse graph → DFS in finish order
📈Dynamic Programming — All Patterns
  • ★1D DP: house robber — max(dp[i-2]+nums[i], dp[i-1])
  • ★0/1 Knapsack: dp[i][w] = max(skip, take if w>=wi)
  • ★Unbounded knapsack: dp[w] = max over all items fitting
  • ★LCS: dp[i][j] = dp[i-1][j-1]+1 if match, else max skip
  • ★LIS O(nlogn): patience sort with binary search
  • →Edit distance: dp[i][j] = min(insert,delete,replace)
  • →Coin change min: dp[i] = min(dp[i-coin]+1) for each coin
  • →Coin change count: dp[i] += dp[i-coin] for each coin
  • →Unique paths grid: dp[i][j] = dp[i-1][j]+dp[i][j-1]
  • →Palindrome partition: dp[i][j]=true if s[i..j] palindrome
  • →Bitmask DP TSP: dp[mask][i] = min cost visiting mask ending i
  • →Stock problems: state machine — hold/sell/cooldown/tx
  • →Word break: dp[i]=true if dp[j] and s[j..i] in dict
  • →Interval DP: dp[i][j] = min cost to solve subproblem i..j

🔍Indexing + Query Optimization
  • ★B-Tree: balanced tree, O(logn) search, good for range queries
  • ★Composite index: (a,b,c) helps queries on (a), (a,b), (a,b,c) — left prefix

  • →Hash index: O(1) equality only, no range, no sort
  • →Partial index: WHERE clause on index, smaller and faster
  • →Clustered vs non-clustered: clustered = data sorted by index
  • →Index selectivity: low cardinality (boolean) = bad candidate
  • →Function on indexed col breaks index: WHERE YEAR(date) = 2024
  • →Index overhead: every INSERT/UPDATE must update all indexes

🔐Transactions & ACID

  • →Two-phase locking: growing phase (acquire) + shrinking (release)
  • →MVCC: readers don't block writers (Postgres default)

🌱 System Design: Absolute Basics (Start Here)

If you don't know these cold, no amount of advanced SD will help. Master every concept here before moving forward.

● LEVEL 0 — ABSOLUTE BEGINNER
🖥️What Happens When You Type google.com
  • 1Browser checks local DNS cache — if found, done
  • 2Query OS resolver → ISP DNS resolver → Root DNS server
  • 3Root DNS → TLD server (.com) → Authoritative DNS → returns IP
  • 4Browser opens TCP connection to IP:443 — 3-way handshake (SYN, SYN-ACK, ACK)
  • 5TLS handshake: exchange certs, agree on cipher, establish session keys
  • 6Browser sends HTTP GET request over encrypted TLS tunnel
  • 7Request hits CDN edge server — if cached, return immediately
  • 8If not cached: CDN forwards to origin Load Balancer
  • 9Load Balancer picks a server (round-robin/least-conn) and forwards
  • 10Web server processes, queries database, returns HTML response
  • KeyEvery step above is a potential failure point — design for each
🌐Networking Fundamentals You Must Know
  • IPIP Address: unique identifier for every device on a network
  • TCPTCP: reliable, ordered, connection-based — 3-way handshake, retransmit on loss
  • UDPUDP: fast, unreliable, no connection — video streaming, gaming, DNS
  • HTTPHTTP/1.1: text protocol, one request per connection (keep-alive)
  • H2HTTP/2: binary, multiplexed streams, header compression, server push
  • H3HTTP/3: QUIC (UDP-based), no head-of-line blocking, 0-RTT
  • WSWebSocket: full-duplex over single TCP connection — chat, real-time feeds
  • SSLTLS: asymmetric key exchange → symmetric encryption for session
  • DNSDNS TTL: how long a DNS record is cached — 300s = 5 min
  • CDNCDN: geographically distributed servers cache static assets near users
  • KeyLatency numbers: L1 1ns | RAM 100ns | SSD 100μs | Net round-trip 100ms
📦Client-Server Architecture
  • 1Client: makes requests — browser, mobile app, IoT device
  • 2Server: processes requests, runs business logic, returns responses
  • 3Database: persistent storage for server — SQL or NoSQL
  • 4Cache: fast in-memory layer between server and DB — Redis
  • 5Load Balancer: distributes traffic across multiple server instances
  • 6Message Queue: async communication between services — Kafka, RabbitMQ
  • 7CDN: serves static files (images, JS, CSS) from edge near user
  • WhySingle server = single point of failure. Scale everything horizontally.
  • RuleStateless servers = can scale horizontally. Session in Redis, not server RAM.
📏Back-of-Envelope Estimation (MUST master this)
  • 1DAU to QPS: 10M DAU × 10 requests/day ÷ 86400s = ~1,160 QPS
  • 2Peak QPS: average QPS × 2–3 (peak multiplier)
  • 3Storage per day: 1M writes/day × 1KB/write = 1GB/day
  • 4Storage 5 years: 1GB/day × 365 × 5 = ~1.8TB
  • 5Bandwidth: 1M req/s × 100KB avg = 100GB/s
  • 6Memory for cache: 1M active users × 1KB session = 1GB Redis
  • TrickPowers of 2: 2^10=1K, 2^20=1M, 2^30=1B. Memorize these.
  • Trick1 million seconds ≈ 11.5 days. 1 billion seconds ≈ 31.7 years.
  • RuleAlways do this in first 5 minutes of SD interview. Shows senior thinking.

SD MODULE 03

⚡ Caching & Databases at Scale (Month 4–5)

Caching is the most powerful weapon in system design. Database scaling is the hardest problem. Master both completely.

● LEVEL 2 — INTERMEDIATE
⚡Caching — Every Strategy Explained
  • CACache Aside (Lazy): App reads cache → miss → reads DB → populates cache
  • PROOnly cache what's actually requested. Cache failures don't break reads.
  • CONCache miss = 3 operations. Initial requests are slow. Stale data risk.
  • WTWrite Through: App writes to cache AND DB synchronously
  • PROCache always fresh. No stale reads after writes.
  • CONWrite latency doubles. Cache gets polluted with write-only data.
  • WBWrite Behind (Back): App writes cache only, async flush to DB
  • PROFastest writes. Batching DB writes is efficient.
  • CONData loss risk if cache dies before flush. Complex to implement.
  • RTRead Through: Cache sits in front, fetches from DB on miss automatically
  • CPCache Penetration: query for non-existent key → always hits DB
  • FIXCache null result with short TTL. Use Bloom filter before cache.
  • CSCache Stampede: cache expires → 1000 threads all miss → DB overwhelmed
  • FIXMutex: only one thread fetches, others wait. Or probabilistic early expiry.
  • CA2Cache Avalanche: many keys expire at same time → DB flood
  • FIXAdd random jitter to TTL: TTL = 300 + random(0, 60)
🗄️Database Scaling — All Techniques
  • 1Vertical Scaling: bigger server (more CPU/RAM). Simple but has ceiling.
  • 2Read Replicas: primary handles writes, replicas handle reads. Simple read scale.
  • 3Connection Pooling: HikariCP reuses DB connections. Max pool = CPU*2+disk.
  • 4Query Optimization: add indexes, fix N+1, rewrite slow queries first.
  • 5Caching Layer: Redis in front of DB — most impactful optimization.
  • 6Horizontal Partitioning (Sharding): split rows across multiple DBs
  • SHHash sharding: shard = hash(userId) % N. Even distribution. Hard to add shards.
  • SHRange sharding: user 1-1M → shard1, 1M-2M → shard2. Easy range queries. Hot spots.
  • SHGeo sharding: US→shard1, EU→shard2. Low latency by region.
  • CHConsistent Hashing: ring topology, add/remove nodes = minimal key remapping
  • VNVirtual Nodes: each physical node owns multiple ring positions = balanced load
  • RULRule: always add read replicas + cache before sharding. Sharding is last resort.
🆚SQL vs NoSQL — When to Use What
  • SQLUse SQL when: complex joins needed, ACID transactions required, schema is stable
  • EXUser accounts, financial transactions, order management, inventory
  • NSUse NoSQL (Document) when: flexible schema, nested/hierarchical data
  • EXUser profiles, product catalogs, CMS content — MongoDB
  • KVUse Key-Value store when: simple get/set by key, extremely high throughput
  • EXSessions, cache, leaderboards — Redis, DynamoDB
  • COLUse Wide-Column when: time-series data, write-heavy, massive scale
  • EXIoT metrics, activity logs, messaging history — Cassandra, HBase
  • GRUse Graph DB when: data is a network (relationships are first-class)
  • EXSocial networks, fraud detection, recommendations — Neo4j
  • TIPMost real systems use polyglot persistence — multiple DB types
🏛️CAP Theorem — Deep Understanding
  • CConsistency: every read gets the most recent write (or error)
  • AAvailability: every request gets a response (may not be latest data)
  • PPartition Tolerance: system works even when network splits occur
  • KEYP is always required — network partitions WILL happen. Choose C or A.
  • CPCP Systems: consistent but may be unavailable during partition

🔒Consistency Models — Deep Dive
  • SCStrong Consistency: every read sees latest write. Requires coordination. Slow.
  • EXBank balance, stock inventory — correctness over speed.
  • ECEventual Consistency: replicas converge eventually. Reads may be stale. Fast.
  • EXSocial media likes count, DNS propagation, shopping cart.
  • CACausal Consistency: if A causes B, all see A before B. Stronger than eventual.
  • EXComments on a post — reply must appear after parent comment.
  • MRMonotonic Read: once you read a value, never see older value from same client.
  • RYWRead Your Writes: you always see your own writes immediately.
  • RULERule: start with eventual consistency, strengthen only where correctness critical.
  • CONFConflict resolution: Last-Write-Wins (LWW), vector clocks, CRDTs.
🤝Consensus & Leader Election
  • PROBProblem: distributed nodes need to agree on a single value. Network can fail.
  • RFRaft: leader-based consensus. Leader replicates log to followers. Majority quorum.
  • RFLeader election: node with most up-to-date log + majority votes wins.
  • USEUsed in: etcd, CockroachDB, TiKV, Consul.
  • PXPaxos: classic consensus. Proposer → Acceptors → Learners. More complex than Raft.
  • ZKZooKeeper: distributed coordination. Ephemeral znodes for leader election.
  • ZKAll nodes try to create /election/leader. First success = leader. Others watch.
  • QUQuorum: W+R > N for strong consistency. Typical: N=3, W=2, R=2.
  • RULEYou don't implement Raft. You use etcd/ZooKeeper. But understand why.
🛡️Resilience Patterns — Production Must-Know
  • CBCircuit Breaker: CLOSED→OPEN on failures→HALF-OPEN after timeout→test→CLOSED
  • CBResilience4j: @CircuitBreaker(name="payment", fallbackMethod="payFallback")
  • RTRetry: retry on transient failures. Use exponential backoff: wait=2^attempt * base
  • JTJitter: add random delay to retry: wait + random(0, wait). Prevents thundering herd.
  • TOTimeout: every network call MUST have timeout. No timeout = thread leak on failure.
  • TOSet at multiple levels: connection timeout, read timeout, request timeout.
  • BHBulkhead: isolate failures. Separate thread pools per downstream service.
  • BHPayment service slow? Bulkhead prevents it from exhausting shared thread pool.
  • FBFallback: return cached data, default response, or partial result on failure.
  • DGGraceful degradation: disable non-critical features first. Core flow must survive.
  • RULEIn interviews: always mention all 5 — CB + Retry + Timeout + Bulkhead + Fallback
🔄CQRS + Event Sourcing
  • CQCQRS: Command Query Responsibility Segregation. Separate read and write models.
  • CQWrite side: normalized DB, strong consistency. Read side: denormalized, optimized for queries.
  • CQSync via events: write DB → publish event → update read DB/cache.
  • ESEvent Sourcing: store events not current state. Replay events to rebuild state.
  • ESAccountCreated → MoneyDeposited → MoneyWithdrawn → balance = replay all
  • ESBenefits: full audit log, time travel (rewind state), event-driven naturally.
  • SGSaga Pattern: distributed transaction via sequence of local transactions + events.
  • CHChoreography: services react to events independently. Decentralized. Hard to debug.
  • OROrchestration: central saga orchestrator tells each service what to do. Easier to trace.
  • COMPCompensating transaction: on saga failure, execute reverse actions (refund, unlock).

  • →Mono.fromCallable(): wrap blocking code safely
  • →Flux.interval(), Flux.range(), Flux.fromIterable()
  • →map() vs flatMap(): map transforms, flatMap merges inner publishers
  • →concatMap(): sequential flatMap — preserves order
  • →zipWith(), zip(): combine multiple publishers element by element
  • →mergeWith(): merge two Fluxes interleaved by arrival time
  • →switchIfEmpty(): fallback when publisher is empty
  • →onErrorResume(): recover from error with fallback publisher
  • →retry(3): retry on error up to N times
  • →publishOn(Schedulers.boundedElastic()): switch thread for blocking ops
  • →subscribeOn(): affects the subscription thread (upstream)
  • →WebFlux: Spring reactive web — non-blocking from controller to DB
  • →R2DBC: reactive DB driver for PostgreSQL, MySQL
